How HTTPS Works

From the moment you press Enter to a fully rendered, encrypted page — every step explained.

Request Lifecycle Overview
1

DNS Resolution

First, the browser needs to turn the human-readable domain name (like example.com) into an IP address that computers use to route traffic. It asks a DNS resolver, which may check its cache, then query a root server, a top-level-domain server, and finally the authoritative name server for the domain. The resolver returns the IP address to the browser. This can involve four or more round trips if nothing is cached.

2

TCP Connection

With the IP address known, the browser opens a TCP connection to the server on port 443. This uses a three-way handshake: the client sends a SYN packet, the server replies with a SYN-ACK packet, and the client responds with an ACK packet. Only after these three messages is the connection established and ready to carry data.

3

TLS Handshake

Because HTTPS is encrypted, the client and server must agree on encryption keys before sending any real data. The client sends a ClientHello listing supported cipher suites. The server responds with a ServerHello choosing a cipher, and sends its certificate. The client verifies the certificate against trusted authorities, then both sides perform a key exchange to derive a shared secret. Finally each side sends a Finished message, and the encrypted channel is ready.

4

HTTP Request & Response

Now the browser sends an encrypted HTTP request: a method (GET), a path, and headers. The server processes it, perhaps querying a database or cache, and returns a response with a status code (like 200 OK), headers, and the HTML body. The browser then parses the HTML, requests additional resources like CSS, JavaScript, and images, and renders the page.

5

Caching

To make future visits faster, responses can be cached at several layers: the browser cache, a CDN edge node, and the server. Cache headers like Cache-Control and ETag tell each layer how long to store a resource and how to check whether it has changed. A well-cached site can skip most of the earlier steps entirely on repeat visits.
